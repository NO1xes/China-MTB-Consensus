# 仓库维护与修订规范

本文件用于规范《中国山地车运动安全与公共行为共识》仓库的维护方式。

目标：

- 保持文本长期稳定性
- 防止结构漂移
- 保证修订可追溯
- 降低情绪化修改风险
- 支持未来多人协作

---

# 一、基本原则

1. 安全优先
2. 合法为本
3. 原则稳定，场景可扩展
4. 不因舆论波动频繁改动核心立场
5. 所有修改必须可解释

---

# 二、版本编号规则

采用语义化版本（简化版）：

v主版本.次版本.修订号

示例：

v0.1.0  初始草案  
v0.2.0  新增场景模块  
v0.2.1  修正文案措辞  
v1.0.0  第一正式版  

规则：

- 主版本变更：原则或结构重大调整
- 次版本变更：新增章节或场景
- 修订号变更：文字修改或表述优化

---

# 三、revision_history.md 记录规范

文件路径：

/docs/appendix/revision_history.md

格式如下：

---

## v0.2.0 — 2026-02-17

### 新增
- 新增“楼梯骑行”场景分级文件
- 新增风险矩阵模型

### 修改
- 调整“Trail 建设”表述措辞

### 说明
本次更新旨在完善场景分级结构。

---

规则：

- 每次发布版本必须记录
- 不记录临时草稿修改
- 仅记录合并到 main 分支的修改

---

# 四、单人维护规范

当仓库为单人维护时：

1. 不直接在 main 分支大规模修改
2. 建议建立 develop 分支进行草稿更新
3. 修改完成后再合并至 main

基本流程：

- 新建分支
- 修改内容
- 自行审阅 24 小时
- 再合并到 main

目的：

避免情绪化即时修改。

---

# 五、多人协作规范

## 1. 分支管理

建议采用简化模型：

- main：稳定版本
- develop：开发版本
- feature/xxx：单个专题修改

禁止：

- 直接向 main 提交大规模修改

---

## 2. Pull Request 规则

每次 PR 必须说明：

- 修改原因
- 修改范围
- 是否影响核心原则
- 是否需要版本号升级

PR 至少 1 人审阅（若多人协作）。

---

## 3. 修改边界

以下内容不得随意修改：

- 核心原则顺序
- 安全优先地位
- 合法为本立场

如需修改，必须：

- 升级主版本号
- 写明理由

---

# 六、Issue 管理规范

Issue 分类建议：

- 讨论（Discussion）
- 建议（Proposal）
- 错误修正（Correction）
- 场景新增（Scenario）
- 法律更新（Legal Update）

处理原则：

- 不处理情绪性攻击
- 不参与对抗性争论
- 不进行意识形态辩论

---

# 七、修改频次建议

核心文本（core）：

建议半年至一年集中修订一次。

场景文件（scenarios）：

可按需更新。

法律解释部分：

仅在政策变化时更新。

---

# 八、文本修改原则

1. 原则不因个案改变
2. 不因单次事故立即扩大限制
3. 不因个别舆论压力转向
4. 保持长期视角

---

# 九、发布节奏建议

v0.x.x

v1.0.0

未来升级：

- 原则稳定后进入低频更新阶段
